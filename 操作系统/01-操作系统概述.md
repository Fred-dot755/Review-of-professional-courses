# 第1章：操作系统概述

> 操作系统是配置在计算机硬件上的第一层软件，是对硬件系统的首次扩充。它管理计算机系统资源，控制程序执行，改善人机界面，为应用软件提供支持。

---

## 1.1 操作系统的定义与目标

### 1.1.1 什么是操作系统？

```
┌─────────────────────────────────────────────┐
│                 用  户                       │
├─────────────────────────────────────────────┤
│              应用程序（浏览器/VSCode等）       │
├─────────────────────────────────────────────┤
│           ┌─────────────────────┐            │
│           │     操作系统(OS)     │            │
│           │  资源管理 + 服务提供  │            │
│           └─────────┬───────────┘            │
├─────────────────────┼───────────────────────┤
│              裸机（硬件层）                    │
└─────────────────────────────────────────────┘
```

**操作系统（Operating System，OS）** 是：
- 管理和控制计算机硬件与软件资源的**系统软件**
- 用户与计算机硬件之间的**接口**
- 计算机系统资源的**管理者**
- 扩充机器（虚拟机）的**实现者**

### 1.1.2 操作系统的四大目标

| 目标 | 说明 |
|------|------|
| **方便性** | 提供用户接口，屏蔽硬件细节，用户无需直接操作硬件 |
| **有效性** | 提高CPU利用率、I/O设备利用率，有效管理和分配资源 |
| **可扩展性** | 便于增加新功能和模块，适应硬件发展（微内核、模块化） |
| **开放性** | 遵循国际标准（如POSIX），实现不同系统间的互操作性 |

## 1.2 操作系统的发展历程 ⭐

### 1.2.1 无操作系统时代（人工操作方式）

- 用户独占全机，CPU等待人工操作
- **人机矛盾**：CPU速度快，人工操作慢
- 输入方式：插拔线路板 → 纸带/卡片

### 1.2.2 单道批处理系统

```
用户程序 ──→ 输入机 ──→ 磁带 ──→ CPU执行 ──→ 磁带 ──→ 输出机 ──→ 结果
```

| 特征 | 说明 |
|------|------|
| **单道性** | 内存中只有一道程序，CPU只为一个程序服务 |
| **顺序性** | 各作业按顺序进入内存并执行 |
| **自动性** | 由**监督程序**自动控制作业流转 |

**主要问题**：CPU与I/O设备速度不匹配，CPU大量时间空闲等待I/O。

### 1.2.3 多道批处理系统 ⭐

```
内存：
┌─────────┐  ┌─────────┐  ┌─────────┐
│ 程序 A   │  │ 程序 B   │  │ 程序 C   │  多道程序同时驻留内存
└─────────┘  └─────────┘  └─────────┘

CPU执行：
  A计算 → B计算 → A I/O(CPU空闲) → C计算 → B I/O(CPU空闲) → ...
```

| 特征 | 说明 |
|------|------|
| **多道性** | 内存中同时存放多道程序 |
| **宏观上并行** | 多道程序同时运行（宏观角度） |
| **微观上串行** | 单CPU交替执行各程序 |
| **无序性** | 作业完成顺序与进入顺序未必一致 |

**核心思想**：当一道程序因I/O等待时，CPU立即切换到另一道程序继续执行，**提高CPU利用率**。

> ⭐ **推动多道批处理系统发展的关键动力**：CPU与I/O设备速度不匹配的矛盾。

### 1.2.4 分时系统（Time-Sharing System）

分时系统是最早的人机交互操作系统类型。

| 特征 | 说明 |
|------|------|
| **多路性** | 多个用户通过终端同时使用一台计算机 |
| **独立性** | 每个用户感觉独占计算机 |
| **及时性** | 用户请求能在可接受时间内得到响应 |
| **交互性** | 用户能与系统进行人机对话 |

**实现原理**：**时间片轮转**——CPU时间被分成很短的时间片，轮流分配给各终端用户。

```
时间片轮转示意图：

终端1: [■□□□] [□■□□] [□□■□] [□□□■] ...
终端2: [□■□□] [□□■□] [□□□■] [■□□□] ...
终端3: [□□■□] [□□□■] [■□□□] [□■□□] ...
终端4: [□□□■] [■□□□] [□■□□] [□□■□] ...
       时间片1  时间片2  时间片3  时间片4
```

> **分时系统 vs 多道批处理系统**：分时系统追求**及时响应**（交互性），多道批处理追求**资源利用率**（吞吐量）。

### 1.2.5 实时系统（Real-Time System）

| 类型 | 应用场景 | 要求 |
|------|----------|------|
| **硬实时系统** | 导弹制导、自动驾驶 | 必须在**绝对严格**的截止时间内完成 |
| **软实时系统** | 视频播放、网络电话 | 偶尔超时可以容忍 |

**特征**：实时性、可靠性 > 资源利用率

### 1.2.6 微机操作系统发展

| 阶段 | 代表系统 | 特征 |
|------|----------|------|
| 单用户单任务 | CP/M、DOS | 一个用户一个任务 |
| 单用户多任务 | Windows 95/98 | 一个用户多个任务并发 |
| 多用户多任务 | UNIX、Linux、Windows NT | 多个用户多个任务并发 |

## 1.3 操作系统的基本特征 ⭐

| 特征 | 含义 | 关键技术 |
|------|------|----------|
| **并发（Concurrency）** | 多个事件在**同一时间间隔**内发生 | 进程、线程、调度 |
| **共享（Sharing）** | 资源被多个并发执行的进程共同使用 | 互斥访问、同步机制 |
| **虚拟（Virtual）** | 将物理实体映射为多个逻辑对应物 | 虚拟内存（时分复用）、虚拟磁盘 |
| **异步（Asynchrony）** | 进程以不可预知的速度推进 | 信号量、管程、消息传递 |

> ⚠️ **并发 vs 并行**：
> - **并发**：宏观上同时，微观上交替执行（单CPU）
> - **并行**：微观上也同时执行（多CPU/多核）

### 并发与共享的关系

```
并发 ──→ 引发资源共享需求
共享 ──→ 需要协调并发访问
二者互为条件、互为存在前提
```

## 1.4 操作系统的主要功能 ⭐

### 1.4.1 五大管理功能

```
                        操作系统
                            │
    ┌───────┬───────┬───────┼───────┬───────┐
    ▼       ▼       ▼       ▼       ▼       ▼
处理器管理 存储器管理 设备管理 文件管理 作业管理 用户接口
```

### 1.4.2 功能详解

#### (1) 处理机管理（进程管理）

| 子功能 | 说明 |
|--------|------|
| **进程控制** | 创建/撤销进程，控制进程状态转换 |
| **进程同步** | 协调进程间的执行次序与资源访问 |
| **进程通信** | 实现进程间的信息交换 |
| **调度** | 作业调度、进程调度 |

#### (2) 存储器管理

| 子功能 | 说明 |
|--------|------|
| **内存分配** | 为程序分配内存空间 |
| **内存保护** | 防止程序越界访问其他程序空间 |
| **地址映射** | 逻辑地址 → 物理地址 |
| **内存扩充** | 虚拟存储器技术 |

#### (3) 设备管理

| 子功能 | 说明 |
|--------|------|
| **缓冲管理** | 缓解CPU与I/O速度差异 |
| **设备分配** | 为进程分配所需设备 |
| **设备驱动** | 隐藏硬件差异，提供统一接口 |
| **虚拟设备** | SPOOLing技术 |

#### (4) 文件管理

| 子功能 | 说明 |
|--------|------|
| **文件存储空间管理** | 管理磁盘空闲空间 |
| **目录管理** | 文件索引与查找 |
| **读写管理** | 文件读写与保护 |

#### (5) 作业管理

- **作业**：用户要求计算机完成的一个任务
- 包括作业调度、作业控制

## 1.5 用户接口

### 1.5.1 三种接口类型

```
                    ┌──────────────────┐
                    │     用    户      │
                    └───────┬──────────┘
                            │
          ┌─────────────────┼─────────────────┐
          ▼                 ▼                 ▼
    ┌──────────┐     ┌──────────┐     ┌──────────────┐
    │ 命令接口  │     │ 程序接口  │     │ 图形用户接口  │
    │ (Shell)  │     │(系统调用)  │     │   (GUI)      │
    └──────────┘     └──────────┘     └──────────────┘
```

| 接口类型 | 使用方式 | 示例 |
|----------|----------|------|
| **命令接口** | 用户在终端输入命令 | `ls -la`、`gcc test.c` |
| **程序接口（系统调用）** | 程序通过API调用OS服务 | `fork()`、`read()`、`open()` |
| **图形用户接口（GUI）** | 图形化操作 | 点击图标、拖拽文件 |

### 1.5.2 系统调用 ⭐

```
用户程序（用户态）
    │
    │ fork()            ← C库函数封装
    ▼
系统调用接口（陷入指令 int 0x80 / syscall）
    │
    ▼
操作系统内核（核心态/内核态）
```

| 概念 | 说明 |
|------|------|
| **系统调用** | 用户程序请求操作系统服务的唯一入口 |
| **陷入（trap）** | 用户态 → 核心态的切换机制 |
| **访管指令** | 触发系统调用的特权指令 |

> ⭐ **系统调用 vs 过程调用**：
> - 系统调用：运行在核心态，通过陷入指令切换；过程调用：运行在用户态，直接跳转。
> - 系统调用：可能引起进程切换（如I/O）；过程调用：不切换进程。

## 1.6 操作系统结构 ⭐

### 1.6.1 四种结构对比

| 结构 | 特点 | 优点 | 缺点 | 代表系统 |
|------|------|------|------|----------|
| **整体式（单内核）** | 所有功能模块集中在一起 | 效率高、通信开销小 | 结构混乱、难以维护 | 早期UNIX、Linux |
| **层次式** | 按功能分层，每层仅使用下层服务 | 结构清晰、易于调试 | 效率低、层间通信开销大 | THE系统 |
| **微内核** | 内核极小，大部分服务在用户态 | 可扩展、可移植、安全 | 效率低、IPC开销大 | Minix、QNX、鸿蒙 |
| **模块化** | 内核为基本模块集，运行时动态加载 | 兼顾效率和可扩展性 | — | 现代Linux、Solaris |

### 1.6.2 微内核 vs 单内核

```
单内核（Monolithic Kernel）：
┌───────────────────────────────────────┐
│            内核态                      │
│  进程管理│内存管理│文件系统│设备驱动│网络  │
└───────────────────────────────────────┘

微内核（Microkernel）：
┌─────────┐   ┌─────────┐   ┌─────────┐
│ 文件系统  │   │ 设备驱动  │   │ 网络协议  │  ← 用户态服务
│ (用户态)  │   │ (用户态)  │   │ (用户态)  │
└────┬─────┘   └────┬─────┘   └────┬─────┘
     └───────────────┼───────────────┘
              ┌──────┴──────┐
              │   微内核      │  ← 最小内核
              │ IPC│调度│中断 │
              └─────────────┘
```

| 对比项 | 单内核 | 微内核 |
|--------|--------|--------|
| **内核大小** | 大（包含所有服务） | 小（仅基本功能） |
| **运行效率** | 高 | 低（IPC开销大） |
| **可扩展性** | 差 | 好 |
| **安全可靠性** | 一般（一个模块崩溃影响全局） | 高（服务隔离） |
| **典型系统** | Linux、Windows（混合） | QNX、Minix、HarmonyOS |

## 1.7 虚拟机

### 1.7.1 虚拟机概念

> 虚拟机（Virtual Machine，VM）是通过软件模拟的具有完整硬件系统功能的计算机系统。

```
┌─────────────────────────────┐
│  应用程序  │  应用程序       │
├────────────┼───────────────┤
│  Guest OS1 │  Guest OS2    │
├────────────┴───────────────┤
│  虚拟机监控器（VMM/Hypervisor）│
├─────────────────────────────┤
│      物理硬件                │
└─────────────────────────────┘
```

### 1.7.2 虚拟机分类

| 类型 | 说明 | 代表 |
|------|------|------|
| **Type 1（裸机型）** | VMM直接运行在硬件上 | VMware ESXi、KVM |
| **Type 2（宿主型）** | VMM运行在宿主OS上 | VMware Workstation、VirtualBox |

## 1.8 操作系统运行环境

### 1.8.1 CPU状态

| 状态 | 特征 | 可执行指令 |
|------|------|-----------|
| **核心态（管态/内核态）** | 操作系统运行的状态 | 特权指令 + 非特权指令 |
| **用户态（目态）** | 用户程序运行的状态 | 仅非特权指令 |

### 1.8.2 特权指令 vs 非特权指令

| 指令类型 | 说明 | 示例 |
|----------|------|------|
| **特权指令** | 只能在核心态执行 | 修改PSW、开关中断、I/O指令 |
| **非特权指令** | 用户态可执行 | 算术运算、逻辑运算、数据传送 |

### 1.8.3 中断与异常 ⭐

中断和异常是操作系统从用户态进入核心态的**唯一途径**。

```
中断/异常分类：
                ┌── 外中断（中断）
                │     ├── 可屏蔽中断（INTR）
                │     └── 不可屏蔽中断（NMI）
                │
中断/异常 ──────┤
                │     ┌── 故障（Fault）   — 可恢复（如缺页）
                └── 内中断（异常）├── 陷阱（Trap）    — 有意安排（系统调用）
                      │
                      └── 终止（Abort）   — 不可恢复（硬件故障）
```

| 对比项 | 外中断（中断） | 内中断（异常） |
|--------|---------------|---------------|
| **来源** | CPU外部 | CPU内部 |
| **与指令关系** | 与当前指令无关 | 由当前指令引发 |
| **响应时机** | 指令执行周期结束时 | 指令执行过程中 |
| **目的** | 处理外部事件 | 处理程序错误或系统调用 |

> ⭐ **中断隐指令**：CPU响应中断时硬件自动完成的操作——关中断、保存断点、取中断服务程序入口地址。这不是真正意义上的指令。
